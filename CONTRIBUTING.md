# Правила разработки Magister

## Держите `structure.txt` актуальным

`structure.txt` — карта структуры проекта. Её нужно обновлять **после каждого
изменения файловой структуры**: добавления, удаления или переименования файла
или папки.

Порядок:

1. Внесите изменения в проект (добавили/удалили/переименовали файлы).
2. Отредактируйте `structure.txt`, отразив изменения (сохраняя стиль и пояснения).
3. Проверьте актуальность:

   ```bash
   npm run structure:check
   ```

Проверка сверяет карту с реальными файлами и завершается ошибкой, если:

- в проекте появилась запись, не описанная в `structure.txt`;
- в `structure.txt` осталась корневая запись, которой уже нет в проекте.

Чтобы не ломаться на однотипных рядах (155 картинок `image-*.jpg`, уроки
`lesson1..24`), поштучно глубокие списки не сверяются — контролируются записи
верхнего уровня и ключевые подкаталоги (`assets/*`, `data/*`). Новые файлы в уже
описанных рядах можно не перечислять поштучно, но новые папки и новые типы
файлов добавлять обязательно.

### Автоматическое напоминание

Проверка запускается автоматически:

- в `npm test` (вместе с юнит-тестами);
- в git-хуке **pre-commit** — коммит не пройдёт, пока `structure.txt` устарел.

Включить хук один раз после клонирования репозитория:

```bash
git config core.hooksPath .githooks
```

Если правку структуры действительно нужно закоммитить без обновления карты
(редкий случай), используйте `git commit --no-verify`.
